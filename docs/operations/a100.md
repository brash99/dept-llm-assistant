# A100 Server Operations

These instructions apply only to the canonical server checkout:

```bash
cd /work/brash/dept-llm-assistant
```

The commands assume the production virtual environment exists at `.venv` and `config/settings.yaml` points to the canonical project root. Do not copy local macOS paths into server configuration.


## Launch and monitor

ISO expects an OpenAI-compatible LLM endpoint configured by `llm.base_url` and `llm.model` in `config/settings.yaml` (currently `http://localhost:8001/v1`). Model-server startup is deployment-specific and is not managed by this repository.

Launch the UI in a dedicated shell or tmux window:

```bash
cd /work/brash/dept-llm-assistant
source .venv/bin/activate
streamlit run web_app.py
```

Useful monitoring commands:

```bash
nvidia-smi
nvidia-smi dmon
tmux ls
tail -f storage/logs/*.log
```

Use separate tmux windows for the model server, Streamlit, GPU monitoring, and long-running ingestion/index jobs. See [tmux reference](../developer/tmux_quick_reference.md).

## Google Drive synchronization

`scripts/sync_drive.sh` runs `rclone sync gdrive: storage/raw_drive`. This can remove local destination files that no longer exist on the remote. Review the remote, destination, and rclone configuration before running it.

```bash
cd /work/brash/dept-llm-assistant
bash scripts/sync_drive.sh
```

Logs are written under `storage/logs/`.

## Inspect and normalize sources

Inventory and list configured normalization sources:

```bash
.venv/bin/python -m scripts.inventory_corpus
.venv/bin/python -m scripts.normalize_documents --list-sources
```

Normalize all configured, available sources:

```bash
.venv/bin/python -m scripts.normalize_documents --source all --limit 1000000
```

Normalize one configured source or one file:

```bash
.venv/bin/python -m scripts.normalize_documents --source cnu_website --limit 1000000
.venv/bin/python -m scripts.normalize_documents --file /absolute/path/to/source.pdf
```

Normalization preserves source boundaries and provenance and writes JSON Knowledge Objects under the configured normalized directory.

## Build the constitutional catalog

The registry is `config/institutional_constitution.yaml`. Its configured source documents must already exist as normalized Knowledge Objects.

```bash
.venv/bin/python -m scripts.build_constitutional_catalog
```

Use `--clear` only when intentionally replacing generated constitutional outputs:

```bash
.venv/bin/python -m scripts.build_constitutional_catalog --clear
```

Constitutional objects encode identified institutional commitments such as the Strategic Compass. They do not prove operational facts or final alignment judgments.

## Chunk, embed, and index

Chunk normalized and constitutional objects:

```bash
.venv/bin/python -m scripts.chunk_documents --limit 1000000
```

Embed chunks using the model/device in `config/settings.yaml`:

```bash
.venv/bin/python -m scripts.embed_chunks --limit 1000000 --embedding-context title_path
```

Build the FAISS index:

```bash
.venv/bin/python -m scripts.build_vector_index
```

The index must be rebuilt after embeddings change. Chunking, embedding, and indexing write derived data; inspect available disk space and logs before large runs.

## Full rebuild warning

`scripts/run_full_pipeline.sh` deletes files from configured derived-output directories before normalization, chunking, embedding, and indexing. It does not rebuild the constitutional catalog between normalization and chunking. For a controlled complete rebuild, run the individual commands above in this order:

1. inventory;
2. normalization;
3. constitutional catalog;
4. chunking;
5. embeddings; and
6. FAISS indexing.

Do not run a destructive rebuild merely to validate documentation or code.

## Retrieval and Decision Brief checks

Question answering invokes the configured LLM:

```bash
.venv/bin/python -m scripts.ask_rag "What evidence describes current faculty capacity?"
```

For retrieval-only inspection, use Developer Mode in Streamlit or a benchmark/failure-analysis command. Raw FAISS and cross-encoder scores are engineering diagnostics, not calibrated executive confidence values.

## Benchmarking

Benchmark files currently live under `benchmarks/retrieval/`. The benchmark script expects its `--benchmark` value relative to `benchmarks/`, so pass the subdirectory explicitly:

```bash
.venv/bin/python -m scripts.run_retrieval_benchmark --benchmark retrieval/v2.yaml
```

Inspect one benchmark case:

```bash
.venv/bin/python -m scripts.analyze_failure \
  --benchmark retrieval/v2.yaml \
  --case CASE_ID \
  --fetch-k 200
```

Benchmark logs are written to `storage/logs/`. The current benchmark serializer records raw, exact-deduped, reranked, thresholded, and final stages; Streamlit Developer Mode additionally exposes document-family diversity.

## Git synchronization

Before integrating remote work:

```bash
git status -sb
git fetch origin
git log --oneline --decorate --graph --max-count=20 --all
```

Pull only from the intended branch and only with a clean or deliberately preserved worktree:

```bash
git pull --ff-only origin "$(git branch --show-current)"
```

Before publishing:

```bash
git status --short
git diff --check
git log -1 --stat
git push origin "$(git branch --show-current)"
```

Do not commit, pull, or push as an incidental part of ingestion or deployment.
